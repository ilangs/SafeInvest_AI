## 3. 데이터 모델 (`stock_terms`)

```sql
id           TEXT PK            -- T001 ~ T230
term         TEXT               -- 표시 용어명 (PER, 코스피...)
term_ko      TEXT               -- 한국어 정식명 (영문 약어 대응)
category     TEXT               -- 17개 카테고리
importance   SMALLINT 1~5       -- 별점
initial_ko   TEXT               -- ㄱ~ㅎ / #
initial_en   TEXT               -- A~Z
tags         TEXT[]
related_ids  TEXT[]             -- 연관 용어 그래프
description  TEXT               -- 정식 정의
easy_desc    TEXT               -- 쉬운 설명 (초보자용)
formula      TEXT               -- 계산식
caution      TEXT               -- 주의사항
view_count   INTEGER            -- 조회수
```

**인덱스**: 초성 / 알파벳 / 카테고리 / 중요도 / **GIN 전문검색**(term + term_ko + description)
**RLS**: 읽기 공개 · 쓰기 인증 사용자만

## 4. 카테고리 체계 (17종)

가치평가 · 재무분석 · 시장기초 · 시장흐름 · 거시경제 · 실적뉴스 · 투자전략 · 위험관리 · 투자심리 · 매매기초 · 차트분석 · 계좌기초 · 투자유의 · 펀드/ETF · 채권/금리상품 · 파생상품 · 연금/은퇴

> 각 카테고리는 고유 컬러를 가지며, 카드/배지/모달 액센트에 일관 적용.

## 5. UI 핵심 기능

| 기능 | 설명 |
| --- | --- |
| 🔍 통합 검색 | term · term_ko · description · easy_desc · tags 모두 매칭 |
| ㄱㄹ **초성 검색** | 한글 자음만 입력 시 자동 초성 모드 (예: `ㄱㄹ` → "금리") |
| 한/영 색인 탭 | ㄱ~ㅎ / A~Z 알파벳 인덱스로 빠른 탐색 |
| 카테고리 필터 | 17색 칩으로 분야 즉시 좁히기 |
| 중요도 정렬 | ★ 5단계 자동 정렬 (핵심필수 → 참고) |
| 카드 뷰 | 카테고리 컬러 라인 + 호버 인터랙션 |
| 상세 모달 | 정의 · 💡 쉽게 이해하기 · 📐 계산식 · ⚠️ 주의사항 · 태그 · 🔗 연관 용어 |
| 연관 용어 그래프 | `related_ids` 기반으로 모달 안에서 클릭 이동 |
| 디바운스 검색 | 300ms 디바운스로 입력 부담 최소화 |
| ESC 닫기 | 키보드 접근성 |

## 6. LLM 큐레이션 파이프라인 (Track B)

```
┌──────────────────────────┐
│ CATEGORY_PLAN            │   카테고리 + 목표 수량 + 힌트 정의
│ (가치평가 15, 재무분석 18…) │
└────────────┬─────────────┘
             ▼
┌──────────────────────────┐
│ 기존 용어 명단 주입       │   중복 차단용 prompt 컨텍스트
└────────────┬─────────────┘
             ▼
┌──────────────────────────┐
│ GPT-4o (temp 0.3)        │   카테고리별 신규 용어 N개 생성
└────────────┬─────────────┘
             ▼
┌──────────────────────────┐
│ new_terms_dryrun.json    │   사람 검수용 (DB 미반영)
└────────────┬─────────────┘
             ▼
     검수 후 upsert_new_terms.py 실행 → Supabase 반영
```

- 기존 95개 → **약 235개로 확장**
- DryRun 분리로 **품질 게이트** 확보 (운영 안정성)

## 8. 의사결정 포인트

| 결정 | 이유 |
| --- | --- |
| 클라이언트에서 직접 Supabase 조회 (백엔드 라우터 미경유) | RLS로 읽기 공개 + 단순 SELECT라 백엔드 경유 불필요 → 응답 속도/구현 비용 절감 |
| 초성 검색 클라이언트 처리 | 230건 규모는 메모리 필터가 충분히 빠름. DB 왕복 없음 |
| LLM 큐레이션 + DryRun 2단계 | LLM 환각 방지, 사람 검수로 품질 보장 |
| `related_ids` 배열 칼럼 | 관계 테이블 없이 그래프 표현 (단순/충분) |

## 9. 발표 시 강조 포인트

1. **콘텐츠 양 + 품질 동시 확보** — LLM 큐레이션으로 95 → 230 확장하면서 DryRun 검수로 품질 유지

5. **확장성** — `extract_terms_by_category.py` 재실행만으로 추후 카테고리/용어 손쉽게 추가
